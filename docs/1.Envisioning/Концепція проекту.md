
# КОНЦЕПЦІЯ ПРОЕКТУ

Етап: *Аналізу проекту (Envisioning)*

Проект: *Розробити веб-застосунок для управління персональними завданнями (Task Manager)*

## **1. НЕОБХІДНІСТЬ ПРОЕКТУ**
### **1.1. Обгрунтування необхідності**
Проєкт спрямований на вирішення проблеми неефективного управління персональними завданнями та відсутності зручних інструментів для їх структурованого збереження і редагування.
На базі інтерв’ю із замовником та обговорень з командою визначено, що користувачі часто стикаються з такими труднощами:
- складно організувати всі задачі в одному місці;
- відсутність можливості зручно форматувати нотатки;
- ризик втрати інформації при відсутності автозбереження;
- незручність використання різних застосунків для створення і перегляду задач;
- відсутність простого способу експорту своїх записів.
  
Наш проект передбачає створення клієнтського веб-застосунку для управління персональними завданнями, який дозволить:
- створювати, редагувати, переглядати та видаляти завдання;
- використовувати Markdown для форматування тексту;
- встановлювати термін виконання;
- відмічати завдання як виконані;
- автоматично зберігати дані;
- експортувати завдання у форматі .md;
- працювати з різних пристроїв через браузер.

Зберігання даних здійснюється локально (localStorage) або з використанням хмарних технологій (Firebase).


### **1.2. Бачення проекту**

Протягом одного місяця планується розробка зручного та інтуїтивного клієнтського веб-застосунку для управління персональними завданнями, орієнтованого на повсякденне використання широкою аудиторією. 
Застосунок забезпечить централізоване створення, редагування та збереження задач із підтримкою Markdown, автозбереженням і можливістю експорту у форматі .md.
Рішення працюватиме без встановлення додаткового програмного забезпечення, буде адаптованим для мобільних пристроїв та підтримуватиме світлу і темну теми інтерфейсу. 
Успішність проєкту визначатиметься можливістю користувача ефективно організовувати свої завдання в одному середовищі без втрати даних та з мінімальними зусиллями.сяць аби повчитись як коротко та містко говорити про складні речі. 


### **1.3. Аналіз вигод**

В результаті впровадження веб-застосунку для управління персональними завданнями, зацікавлені сторони отримають наступні вигоди:

Користувачі (основна аудиторія):
   - можливість зберігати всі завдання в одному місці, що спрощує організацію часу;
   - зручне створення та редагування задач із використанням Markdown;
   - зменшення ризику втрати даних завдяки автозбереженню;
   - доступ до своїх завдань з будь-якого пристрою через браузер;
   - можливість відмічати виконані завдання та контролювати власну продуктивність;
   - швидкий експорт записів у форматі .md для подальшого використання;
   - комфортне використання завдяки підтримці світлої та темної тем.

Замовник:
   - отримання простого та ефективного інструменту для щоденного планування;отримання простого та ефективного інструменту для щоденного планування;
   - вирішення проблеми неструктурованого збереження завдань;
   - можливість подальшого розширення функціоналу за потреби.

Команда розробки:
   - практичний досвід створення клієнтського веб-застосунку;
   - покращення навичок командної роботи та взаємодії із замовником;
   - розвиток навичок роботи з вимогами та їх реалізацією;
   - набуття практичного розуміння всіх етапів процесу розробки програмного забезпечення.
 
## **2. КОНЦЕПЦІЯ РІШЕННЯ**

*Концепція рішення* (solution concept) надає загальний опис підходів, які проектна група планує використовувати для вирішення проблем та/або задоволення вимог зацікавлених сторін.

### **2.1. Цілі та Завдання**

Основні *цілі проекту* зі створення системи обліку навчальних годин (відповідні завдання показані у вигляді списку другого рівня): 
- **спрощення управління персональними завданнями**
   - створення інтерфейсу для додавання нових завдань
   - реалізація можливості редагування та видалення завдань
   - відображення списку всіх завдань в одному місці

- **підвищення зручності роботи з нотатками**
   - підтримка форматування тексту за допомогою Markdown
   - зручний інтерфейс для введення та редагування контенту

- **забезпечення надійного збереження даних**
   - реалізація автозбереження змін
   - запобігання втраті інформації при оновленні сторінки або закритті браузера
 
- **забезпечення доступності застосунку**
   - розробка адаптивного дизайну для мобільних пристроїв та персональних комп’ютерів
   - забезпечення роботи застосунку у браузері без встановлення
   - оптимізація інтерфейсу для різних розмірів екранів
     
- **забезпечення експорту даних**
   - реалізація експорту завдань у форматі .md
   - можливість збереження файлу на пристрої користувача

- **підвищення зручності використання застосунку**
   - реалізація світлої та темної тем інтерфейсу
   - створення простої та зрозумілої навігації для користувача

### **2.2. Припущення і Обмеження**

Основні припущення стосовно функціональності проєкту зі створення веб-застосунку для управління персональними завданнями::
- користувач працює із застосунком через браузер на мобільному пристрої або персональному комп’ютері;
- застосунок використовується для повсякденного планування без потреби у складному функціоналі;
- дані користувача можуть зберігатися локально без обов’язкової реєстрації;
- у разі використання Firebase користувач має доступ до мережі Інтернет;
- автозбереження відбувається автоматично при зміні даних;
- експорт здійснюється виключно у форматі .md, оскільки цього достатньо для потреб користувача;
- застосунок використовується без встановлення додаткового програмного забезпечення.

Обмеження, які слід враховувати:
- функціональність обмежується лише необхідними можливостями для повсякденного використання;
- локальне зберігання даних має обмеження по обсягу;
- застосунок повинен працювати без встановлення;
- підтримка здійснюється для сучасних браузерів;
- інтерфейс має підтримувати світлу та темну теми;
- обмежений термін розробки (близько одного місяця).

### **2.3. Аналіз використання**

#### КОРИСТУВАЧІ

З веб-застосунком для управління персональними завданнями можуть взаємодіяти користувачі різних категорій, оскільки він орієнтований на широку аудиторію та повсякденне використання.

З системою обліку навчальних годин будуть взаємодіяти наступні групи користувачів:
- учні та студенти, які використовують застосунок для організації навчальних завдань;
- користувачі, які організовують особисті справи (побутові задачі, щоденні плани);
- будь-які інші користувачі, зацікавлені у простому інструменті для планування та ведення нотаток.

### **2.4. Вимоги**

#### ВИМОГИ КОРИСТУВАЧІВ

Основні вимоги до функціональності веб-застосунку для управління персональними завданнями виглядають наступним чином:
- створення, редагування та видалення завдань;
- перегляд усіх завдань в одному інтерфейсі;
- можливість позначати завдання як виконані;
- автоматичне збереження змін без участі користувача;
- доступ до застосунку через браузер без встановлення;
- підтримка світлої та темної тем інтерфейсу;
- зручний та інтуїтивно зрозумілий інтерфейс для повсякденного використання;
- можливість використання застосунку на мобільних пристроях.

#### СИСТЕМНІ ВИМОГИ

Системні вимоги до продукту: 
- застосунок повинен працювати у веб-браузері без встановлення додаткового програмного забезпечення;
- підтримка сучасних браузерів (Google Chrome, Mozilla Firefox, Microsoft Edge тощо); 
- адаптивний інтерфейс для мобільних пристроїв та персональних комп’ютерів;
- швидке завантаження та стабільна робота застосунку;
- забезпечення автозбереження даних;
- мінімальне використання серверних ресурсів;
- коректне відображення інтерфейсу у світлій та темній темах.

### **2.5. Рамки**

#### ФУНКЦІОНАЛЬНІСТЬ РІШЕННЯ

Основні можливості веб-застосунку для управління персональними завданнями:
- управління списком завдань (створення, редагування, видалення);
- перегляд усіх завдань в одному інтерфейсі;
- підтримка Markdown для форматування опису задач;
- можливість відмітки завдань як виконаних;
- автоматичне збереження змін;
- експорт завдань у форматі .md;
- підтримка світлої та темної тем інтерфейсу;
- збереження даних за допомогою localStorage або Firebase.

#### ЗА РАМКАМИ РІШЕННЯ

З огляду на обмежений термін реалізації проєкту та пріоритет простоти використання, за рамки рішення винесено наступні можливості:
- синхронізація даних між різними користувачами;
- push-сповіщення про дедлайни;
- реалізація багатокористувацького режиму (спільний доступ до завдань);
- підтримка додаткових форматів експорту (PDF, DOCX тощо);
- реалізація окремого мобільного застосунку (не лише веб-версія).

### **2.6. Критерії схвалення рішення**

Рішення про готовність веб-застосунку до впровадження замовник буде приймати на основі наступних критеріїв:
- реалізовано всю функціональність, визначену в межах рішення (окрім можливостей, винесених за рамки);
- астосунок забезпечує створення, редагування, видалення та перегляд завдань без помилок;
- коректно працює підтримка Markdown для форматування тексту;
- реалізовано можливість встановлення дедлайнів і відмітки виконаних завдань;
- автозбереження даних працює стабільно без втрати інформації;
- експорт завдань у форматі .md виконується без помилок;
- застосунок стабільно працює у сучасних браузерах;
- інтерфейс коректно відображається на мобільних пристроях і персональних комп’ютерах;
- реалізовано підтримку світлої та темної тем;
- користувач може почати роботу із застосунком без додаткового навчання;
- проєкт завершено у встановлені терміни.

### **3.1. Стратегія архітектурного проектування**

На основі розробленого списку можливостей і функцій формується *стратегія архітектурного проектування* (architectural design strategy), яка описує рішення в цілому. 

Стратегія архітектурного проектування визначає *компоненти* рішення і їх *взаємодію*. Чудовий спосіб опису рішення на цьому етапі - використання ілюструючих діаграм UML (наприклад, **діаграми класів**, **діаграми компонентів**, **діаграма розгортання** тощо).

### **3.2. Стратегія технологічного проектування**

Розробка рішення потребує використання певних *програмних продуктів, бібліотек, фреймворків, технологій*. 

Стратегія технологічного проектування (technical design strategy) описує, які технології і програмні продукти були обрані проектною групою у якості засобу реалізації рішення.

:point_right: Аргументовано опишіть, які технологічні засоби буде використано в процесі роботи над рішенням.

>- мова програмування С#:

>   - ASP.NET Core Web API - для розробки серверної частини системи
> - Bootstrap - для забезпечення адаптивності веб-інтерфейсу
>- хмарна платформа Microsoft Azure:
>   - App Services - для хостингу веб-застосунку
>   - Function App - для хостингу REST-сервісів

>   - Storage Account - для збереження даних звітів 
> - Power BI - для побудови звітів та візуалізацій
