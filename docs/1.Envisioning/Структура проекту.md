## **1. ЦІЛІ ТА ЗАВДАННЯ**

Основні *цілі проекту* зі створення системи обліку навчальних годин:

**Спрощення управління персональними завданнями**:
   - створення інтерфейсу для додавання нових завдань;
   - реалізація можливості редагування та видалення завдань;
   - відображення списку всіх завдань в одному місці.

**Підвищення зручності роботи з нотатками**:
   - підтримка форматування тексту за допомогою Markdown;
   - зручний інтерфейс для введення та редагування контенту.

**Забезпечення надійного збереження даних**:
   - реалізація автозбереження змін;
   - запобігання втраті інформації при оновленні сторінки або закритті браузера.
 
**Забезпечення доступності застосунку**
   - розробка адаптивного дизайну для мобільних пристроїв та персональних комп’ютерів;
   - забезпечення роботи застосунку у браузері без встановлення;
   - оптимізація інтерфейсу для різних розмірів екранів.
     
**Забезпечення експорту та імпорту даних**:
   - реалізація експорту та імпорту завдань у форматі .md;
   - можливість збереження файлу на пристрої користувача.

**Підвищення зручності використання застосунку**:
   - реалізація світлої та темної тем інтерфейсу;
   - створення простої та зрозумілої навігації для користувача.

## **2. ПРИПУЩЕННЯ І ОБМЕЖЕННЯ**

Основні припущення стосовно функціональності проєкту зі створення веб-застосунку для управління персональними завданнями::
- користувач працює із застосунком через браузер на мобільному пристрої або персональному комп’ютері;
- застосунок використовується для повсякденного планування без потреби у складному функціоналі;
- дані користувача можуть зберігатися локально без обов’язкової реєстрації;
- у разі використання Firebase користувач має доступ до мережі Інтернет;
- автозбереження відбувається автоматично при зміні даних;
- експорт здійснюється виключно у форматі .md, оскільки цього достатньо для потреб користувача;
- застосунок використовується без встановлення додаткового програмного забезпечення.

Обмеження, які слід враховувати:
- функціональність обмежується лише необхідними можливостями для повсякденного використання;
- локальне зберігання даних має обмеження по обсягу;
- застосунок повинен працювати без встановлення;
- підтримка здійснюється для сучасних браузерів;
- інтерфейс має підтримувати світлу та темну теми;
- обмежений термін розробки (близько одного місяця).

### **5.2. Керування конфігурацією**

Для забезпечення цілісності проекту та відстеження всіх змін у програмному коді й документації впроваджується стратегія керування конфігурацією на базі системи **Git** та сервісу **GitHub**.

#### **Методи і засоби керування конфігурацією**
Основним інструментом є розподілена система контролю версій **Git**. Весь проектний артефакт (код, SQL-скрипти, ТЗ, UML-діаграми) зберігається у віддаленому репозиторії на **GitHub**.
* **Об'єкти конфігурації:** вихідний код, проектна документація, конфігураційні файли середовища та медіа-ресурси.
* **Методи:** використання моделі галуження (branching), де кожна нова функція розробляється в окремій гілці, та обов'язкове використання **Pull Requests** для контролю якості змін.

#### **Запити та прийняття змін до конфігурації**
Процес внесення змін до проекту регламентований наступними кроками:
1. **Реєстрація:** Створення завдання (**Issue**) на GitHub з описом потреби (фіча або баг).
2. **Розробка:** Виконавець працює у власній гілці (наприклад, `feature/ui-update`).
3. **Обговорення (Pull Request):** Створюється запит на злиття. Команда проводить перегляд коду/документації, залишає коментарі та, за потреби, вносить правки. Коментарі до PR виконують роль протоколу узгодження.
4. **Тестування та злиття:** Після схвалення (Approve) від відповідальних осіб та успішного тестування, зміни зливаються (**Merge**) з основною гілкою проекту.

### **5.4. Керування впровадженнями**

В MSF робота над рішенням не закінчується з закінченням етапу розробки. Важливим аспектом, який вимагає уваги, є *процес впровадження* (розгортання, deployment) як пілотних так і фінальних версій рішення. 

В цьому розділі мають бути описані **способи і засоби** з підготовки релізів та керування їх впровадженням, як в тестові, так і у виробничі середовища.

**Тестування:** перед впровадженням готової версії програмного продукту в тестове або робоче середовище необхідно провести модульне, інтеграційне та комплексне тестування. Це дає змогу перевірити коректність роботи основних функцій, стабільність взаємодії між компонентами системи, загальну працездатність, а також виявити можливі помилки, що можуть вплинути на якість і безпечність подальшого використання продукту.

**Оцінка результатів:** після успішного завершення тестування необхідно виконати оцінку результатів впровадження рішення. Така оцінка передбачає аналіз відповідності реалізованого функціоналу поставленим вимогам, перевірку ефективності роботи системи у цільовому середовищі, визначення впливу рішення на зручність використання та продуктивність роботи користувачів, а також виявлення можливих недоліків і формування пропозицій щодо подальшого вдосконалення продукту.

**Розгортання:** розроблений веб-застосунок потрібно розгорнути на сервері або використати будь-які інструменти хмарного розгортання (накшталт github pages, firebase hosting тощо), щоб зробити його доступним для користувачів через мережу Internet. Апаратні вимоги для користувача і при обраному варіанті розгортання, серверу, мінімальні. Уся інформація оброблюється та зберігається в local storage та firebase.

**Керування проектом:** для успішного впровадження розробленого рішення необхідно забезпечити ефективне управління процесом розгортання. Це включає координацію роботи всіх учасників команди, загальний контроль за виконанням етапів впровадження, забезпечення виконання завдань згідно з планом та регулярне звітування перед замовником.

### **5.5. Досягнення якості проекту**

Цей розділ описує стратегію забезпечення якості веб-застосунку "Task Manager", що базується на запобіганні технічним помилкам та відповідності функціональним вимогам замовника.

#### **Очікування стосовно якості рішення**
Основним очікуванням замовника є реалізація повної запланованої функціональності:
* **Надійність збереження:** коректна робота з *localStorage* та автозбереження без втрати даних (мінімізація ризику №15 та №19).
* **Коректність обробки:** підтримка форматування Markdown та безпомилковий експорт у формат `.md`.
* **Адаптивність:** стабільна робота інтерфейсу на різних типах пристроїв та у різних браузерах (ризик №18).
* **Продуктивність:** швидке завантаження списку завдань та відсутність затримок при редагуванні контенту.

#### **Процес перевірки якості**
Перевірка якості є безперервним процесом і включає наступні види тестування:

| Вид тестування | Об'єкт перевірки | Інструментарій | Хто виконує | Періодичність |
| :--- | :--- | :--- | :--- | :--- |
| **Unit-тестування** | Логіка обробки завдань, методи збереження | **JUnit 5** | Розробники (Левковська, Єдалов) | При кожній збірці модуля |
| **Інтеграційне тестування** | Взаємодія між UI та сервісами обробки даних | **Postman / Mockito** | Розробники (Левковська, Єдалов) | При об'єднанні компонентів |
| **Системне (UI) тестування** | Адаптивність, темна/світла теми, Markdown-редактор | **Selenium / Manual** | Тестувальники (Савеля, Зорій) | Раз на тиждень (стабільна збірка) |
| **Регресійне тестування** | Перевірка того, що нові зміни не пошкодили старий функціонал | **Checklists** | Тестувальники (Савеля, Зорій) | Після виправлення багів |
| **UX-тестування** | Зручність навігації, інтуїтивність інтерфейсу | **Manual review** | UX-спеціаліст (Москов) | Після завершення етапу дизайну |

#### **Процес керування досягненням якості**
Для мінімізації ризиків та досягнення високих стандартів, команда впроваджує такі заходи:
1.  **Code Review:** Перевірка коду колегами перед злиттям у гілку `main` (запобігання ризику №5 "Некваліфіковані співробітники").
2.  **Bug Tracking:** Використання **GitHub Issues** для фіксації та пріоритезації знайдених дефектів.
3.  **Автозбереження та валідація:** Впровадження захисних механізмів у коді для запобігання втрати даних при закритті браузера.
4.  **Аналіз відповідності (Traceability):** Регулярна звірка розробленого функціоналу з переліком цілей у розділі 1.
