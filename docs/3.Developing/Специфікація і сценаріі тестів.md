## **1. ВХІДНІ СПЕЦИФІКАЦІЇ**

Розділ «Вхідні специфікації» регламентує **входи**, необхідні для проходження всіх тестів для даного рішення. У числі таких входів можуть бути дані, введені з клавіатури, натиснення кнопок миші у відповідних місцях сторінки, датчики, комп'ютерне та інше обладнання, різні дії користувачів. 

Опис специфікацій поряд з найменуванням входу містить його *характеристики*, такі як значення (що введено), структури даних (якого типу), інтерфейси (яким чином), протоколи, бази даних, файли тощо.

Для тестування застосунку використовуються такі вхідні специфікації:

| № | Найменування входу | Характеристика | Тип даних | Спосіб введення |
|---|---------------------|----------------|-----------|-----------------|
| 1 | Логін користувача | Ім’я користувача або електронна пошта для входу в систему | Рядок (String) | Введення з клавіатури у форму авторизації |
| 2 | Пароль користувача | Пароль для входу в обліковий запис | Рядок (String) | Введення з клавіатури у форму авторизації |
| 3 | Назва завдання | Коротка назва персонального завдання | Рядок (String) | Введення у текстове поле |
| 4 | Опис завдання | Детальний опис завдання | Текст (Text) | Введення у текстову область |
| 5 | Дата виконання | Запланована дата завершення завдання | Дата (Date) | Вибір у календарі або введення у поле |
| 6 | Пріоритет завдання | Рівень важливості завдання: низький, середній, високий | Перелік значень (Enum/String) | Вибір зі списку |
| 7 | Статус завдання | Поточний стан завдання: нове, в процесі, виконане | Перелік значень (Enum/String) | Вибір зі списку |
| 8 | Пошуковий запит | Текст для пошуку завдань за назвою або описом | Рядок (String) | Введення у поле пошуку |
| 9 | Фільтр завдань | Параметри фільтрації за статусом, пріоритетом або датою | Набір значень | Вибір елементів інтерфейсу |
| 10 | Команда створення | Дія користувача для додавання нового завдання | Подія інтерфейсу | Натискання кнопки «Додати» / «Створити» |
| 11 | Команда редагування | Дія користувача для зміни існуючого завдання | Подія інтерфейсу | Натискання кнопки «Редагувати» |
| 12 | Команда видалення | Дія користувача для видалення завдання | Подія інтерфейсу | Натискання кнопки «Видалити» |
| 13 | Команда збереження | Підтвердження змін після створення або редагування | Подія інтерфейсу | Натискання кнопки «Зберегти» |
| 14 | Команда скасування | Відміна поточної дії без збереження змін | Подія інтерфейсу | Натискання кнопки «Скасувати» |
| 15 | Вибір завдання зі списку | Відкриття конкретного завдання для перегляду або редагування | Ідентифікатор / подія | Натискання на запис у списку |
| 16 | Дані бази даних | Записи про завдання, користувачів, статуси та пріоритети | JSON / таблиці БД | Автоматичне зчитування системою |
| 17 | Дані сесії користувача | Інформація про авторизованого користувача | Токен / службові дані | Формується автоматично після входу |
| 18 | HTTP-запити клієнта | Запити на створення, зміну, видалення та отримання завдань | JSON / HTTP Request | Передаються через веб-інтерфейс |

Розділ «Вихідні специфікації» регламентує **виходи**, очікувані як результат проходження тестів для вашого рішення. Виходи можуть містити дані, інформацію про час виконання операцій, повідомлення, логи, дампи пам'яті, файли, знімки екрану тощо. 

Для тестування застосунку використовуються такі вихідні специфікації:

| № | Найменування виходу | Характеристика | Тип даних | Спосіб отримання |
|---|----------------------|----------------|-----------|------------------|

| 4 | Відображення нового завдання | Поява нового запису після успішного створення | Об’єкт / елемент списку | Оновлення інтерфейсу |
| 5 | Оновлене завдання | Відображення змінених даних після редагування | Об’єкт / елемент списку | Оновлення інтерфейсу |
| 6 | Видалення завдання зі списку | Зникнення запису після підтвердження видалення | Зміна стану списку | Оновлення інтерфейсу |
| 7 | Повідомлення про успішне збереження | Інформування про успішне створення або зміну завдання | Рядок (String) | Спливаюче повідомлення / текст в інтерфейсі |
| 8 | Повідомлення про помилку валідації | Інформація про некоректно заповнені або порожні поля | Рядок (String) | Відображення біля полів або у формі |
| 9 | Результати пошуку | Відфільтрований список завдань відповідно до пошукового запиту | Масив об’єктів / список | Відображення на сторінці |
| 10 | Результати фільтрації | Список завдань, що відповідає обраним параметрам | Масив об’єктів / список | Відображення на сторінці |
| 11 | Зміна статусу завдання | Оновлений стан завдання після вибору нового статусу | Рядок / об’єкт | Оновлення інтерфейсу |
| 12 | Відповідь сервера | Результат обробки HTTP-запиту на клієнтські дії | JSON / HTTP Response | Отримання у клієнтській частині |
| 13 | Код статусу HTTP | Код результату виконання запиту (200, 201, 400, 401, 404, 500 тощо) | Число (Integer) | Отримання у відповіді сервера |

| 15 | Логи виконання | Технічна інформація про запити, помилки, події системи | Текст / журнал | Перегляд у консолі або файлі логів |
| 16 | Час відповіді системи | Тривалість виконання операції або завантаження сторінки | Число / час | Вимірювання інструментами тестування |
| 17 | Візуальні зміни інтерфейсу | Оновлення сторінки, кнопок, повідомлень, полів після дій користувача | Стан UI | Спостереження в браузері |
| 18 | Збережені дані в БД | Фактичне додавання, оновлення або видалення записів у базі даних | Записи БД | Перевірка через сервер або інструменти БД |

## **3. СЕРЕДОВИЩЕ ТЕСТУВАННЯ**

У цьому розділі описуються характеристики та конфігурація середовища, необхідного для проведення тестування веб-застосунку **Task Manager**. Середовище тестування формується з урахуванням обраних методів тестування, а саме: модульного, функціонального, тестування інтерфейсу, тестування зручності використання, сумісності, продуктивності та регресійного тестування.

Оскільки застосунок повинен коректно працювати як на **комп’ютерах і ноутбуках**, так і на **мобільних пристроях**, тестове середовище повинно охоплювати різні типи пристроїв, операційних систем, браузерів та інструментів перевірки.

### **Апаратне середовище**

Для проведення тестування використовуються такі пристрої:

- персональний комп’ютер;
- ноутбук;
- смартфон;
- за можливості планшет.

Мінімальні вимоги до комп’ютерних пристроїв:

- **процесор:** Intel Core i3 / AMD Ryzen 3 або вище;
- **оперативна пам’ять:** не менше 4 ГБ;
- **вільне місце на диску:** не менше 1 ГБ;
- **роздільна здатність екрана:** не менше 1366×768;
- **пристрої введення:** клавіатура, миша або тачпад;
- **мережа:** доступ до Інтернету.

Мінімальні вимоги до мобільних пристроїв:

- **операційна система:** Android або iOS;
- **наявність сучасного браузера** з підтримкою HTML5, CSS3 та JavaScript;
- **сенсорний екран** для перевірки коректності touch-взаємодії;
- **підключення до Інтернету** через Wi-Fi або мобільну мережу.

### **Середовище відповідно до методів тестування**

Для кожного з обраних методів тестування використовується відповідне середовище та інструменти:

- **Модульне тестування** виконується у середовищі розробки для перевірки окремих компонентів системи.
- **Функціональне тестування** проводиться у браузері шляхом перевірки основних сценаріїв роботи застосунку.
- **Тестування інтерфейсу (UI testing)** виконується на різних розмірах екранів для перевірки правильності відображення елементів сторінки.
- **Тестування зручності використання (Usability testing)** проводиться на комп’ютерах і мобільних пристроях для оцінки зручності взаємодії з сайтом.
- **Тестування сумісності (Compatibility testing)** виконується за допомогою різних браузерів та сервісу BrowserStack.
- **Тестування продуктивності (Performance testing)** виконується за допомогою Google Lighthouse.
- **Регресійне тестування (Regression testing)** проводиться після внесення змін до функціоналу із використанням повторних перевірок та автоматизованих інструментів.

---

### **Налаштування тестового середовища**

Перед початком тестування необхідно виконати такі дії:

- розгорнути застосунок на локальному сервері або тестовому середовищі;
- перевірити доступність серверної частини та бази даних;
- створити тестові облікові записи користувачів;
- підготувати тестові дані для роботи із завданнями;
- перевірити коректну роботу сайту на різних роздільних здатностях екрана;
- виконати перевірку адаптивності через інструменти браузера;
- за можливості провести перевірку на реальних мобільних пристроях;
- очистити кеш браузера перед повторними тестами.

---

### **Умови проведення тестування**

Під час перевірки необхідно враховувати:

- коректну роботу сайту на комп’ютерах, ноутбуках і смартфонах;
- правильне відображення сторінок у різних браузерах;
- адаптацію інтерфейсу під різні розміри екрана;
- зручність використання елементів керування мишею та сенсорним введенням;
- правильність виконання операцій створення, редагування, пошуку та видалення завдань;
- стабільність роботи після змін у функціональності;
- швидкість завантаження сторінок і час реакції системи.

## **4. СПЕЦІАЛЬНІ ВИМОГИ**

Для коректного проведення тестування веб-застосунку **NotePro** необхідно дотримуватись таких спеціальних вимог:

1. **Стабільне підключення до мережі Інтернет.**  
   Оскільки застосунок працює через веб-інтерфейс і взаємодіє із серверною частиною, тестування повинно виконуватись за наявності стабільного інтернет-з’єднання. Рекомендована пропускна здатність каналу — не менше **10 Мбіт/с**.

2. **Наявність розгорнутого тестового середовища.**  
   Перед виконанням тестів застосунок має бути розгорнутий на локальному сервері або тестовому хостингу з доступною серверною частиною та базою даних.

3. **Використання тестових облікових записів.**  
   Для перевірки функцій авторизації, створення, редагування, пошуку та видалення завдань повинні бути підготовлені окремі тестові акаунти користувачів.

4. **Попереднє наповнення бази тестовими даними.**  
   Для проведення повноцінного функціонального тестування необхідно створити набір тестових записів із різними назвами, статусами, пріоритетами та датами виконання завдань.

5. **Очищення кешу та скидання стану перед повторними тестами.**  
   Перед повторним запуском тестів рекомендується очищати кеш браузера, cookies та за потреби скидати тестові дані, щоб уникнути впливу попередніх перевірок на результати.

6. **Перевірка на різних типах пристроїв.**  
   Тестування необхідно проводити не лише на персональному комп’ютері, а й на мобільних пристроях для перевірки адаптивності та коректної роботи інтерфейсу.

7. **Контроль часу відповіді системи.**  
   Під час тестування продуктивності слід враховувати, що основні сторінки та операції повинні завантажуватись без помітних затримок для користувача. Для базових дій бажаний час реакції системи — до **2 секунд**, за винятком складних запитів або слабкого з’єднання.

8. **Наявність доступу до інструментів тестування.**  
   Для виконання окремих видів перевірки повинні бути доступні браузерні інструменти розробника, Google Lighthouse, BrowserStack, а також сервіси ручного й автоматизованого тестування, визначені у попередніх розділах.

9. **Безпечне тестування даних.**  
   Під час тестування слід використовувати лише тестові дані та тестові облікові записи, без застосування реальних персональних даних користувачів.

10. **Фіксація результатів тестування.**  
    Результати проходження тестів, виявлені помилки та зауваження повинні документуватися у відповідних тестових сценаріях або таблицях результатів для подальшого аналізу та регресійної перевірки.

## **5. ВЗАЄМОЗАЛЕЖНОСТІ**

### **Основні взаємозв’язки між видами тестування**

1. **Модульне тестування** є базовим етапом перевірки, оскільки воно дозволяє переконатися у правильності роботи окремих компонентів системи. Лише після успішного проходження модульних тестів доцільно переходити до функціонального тестування.

2. **Функціональне тестування** залежить від результатів модульного тестування, тому що перевірка основних сценаріїв роботи застосунку можлива лише за умови коректної роботи його окремих модулів. Якщо базова логіка працює неправильно, результати функціонального тестування не будуть достовірними.

3. **Тестування інтерфейсу (UI testing)** пов’язане з функціональним тестуванням, оскільки елементи інтерфейсу повинні не лише коректно відображатися, а й правильно виконувати свої функції. Тому перевірка зовнішнього вигляду інтерфейсу та перевірка його поведінки виконуються у тісному взаємозв’язку.

4. **Тестування зручності використання (Usability testing)** залежить від функціонального та UI тестування, оскільки оцінювати зручність застосунку можна лише тоді, коли інтерфейс відображається правильно, а основні функції працюють без помилок.

5. **Тестування сумісності (Compatibility testing)** проводиться після базової перевірки функціональності, тому що немає сенсу перевіряти роботу застосунку в різних браузерах і на різних пристроях, якщо його основні функції ще не працюють стабільно навіть в одному середовищі.

6. **Тестування продуктивності (Performance testing)** також залежить від коректної роботи функціоналу. Перед вимірюванням швидкодії потрібно переконатися, що застосунок правильно виконує основні операції, інакше результати продуктивності можуть бути викривленими.

7. **Регресійне тестування** виконується після внесення змін у функціональність або інтерфейс. Воно залежить від результатів попередніх тестів і використовується для перевірки того, що нові зміни не спричинили помилки у вже перевірених частинах системи.

### **Порядок виконання тестів**

Порядок виконання:

1. **Модульне тестування**  
2. **Функціональне тестування**  
3. **Тестування продуктивності (Performance testing)**  
4. **Тестування інтерфейсу (UI testing)**  
5. **Тестування зручності використання (Usability testing)** 
6. **Тестування сумісності (Compatibility testing)**  
7. **Регресійне тестування**

### **Обгрунтування послідовності**

Такий порядок є доцільним, оскільки спочатку потрібно перевірити правильність роботи окремих компонентів, потім — основних функцій системи, після цього — коректність відображення інтерфейсу, зручність користування, роботу в різних середовищах та швидкодію. Регресійне тестування завершує цикл перевірки і виконується після кожного суттєвого оновлення застосунку.

## **6. ТЕСТИ**
Цей розділ присвячений опису конкретних тестів. В кожному з його підпунктів описано тестування певної частини розробленої функціональності. При цьому формулюються такі характеристики як *очікувана поведінка* (що саме перевіряємо), *очікувана швидкодія* (скільки часу має тривати операція), очікувана достовірність (у відсотках), зауваження, які виникли в процесі тестування.

### **6.1. Тест авторизації користувача**

**Коректність входу в систему**

ОЧІКУВАНА ПОВЕДІНКА:  
Після введення правильного логіна та пароля користувач повинен успішно увійти до системи та бути перенаправленим на головну сторінку застосунку або сторінку зі списком завдань.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 2 секунд.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
100%.

ЗАУВАЖЕННЯ:  
Перевіряється для тестового облікового запису з коректними даними.

**Обробка помилкових даних входу**

ОЧІКУВАНА ПОВЕДІНКА:  
У разі введення неправильного логіна або пароля система повинна відобразити повідомлення про помилку та не надавати доступ до функціоналу застосунку.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 2 секунд.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
100%.

ЗАУВАЖЕННЯ:  
Необхідно перевірити варіанти з неправильним логіном, неправильним паролем та порожніми полями.

### **6.2. Тест створення нового завдання**

**Коректність створення завдання**

ОЧІКУВАНА ПОВЕДІНКА:  
Після введення назви, опису, дати, пріоритету та статусу завдання й натискання кнопки збереження нове завдання повинно з’явитися у списку завдань користувача.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 2 секунд.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
100%.

ЗАУВАЖЕННЯ:  
Слід перевірити, чи зберігаються всі введені поля без спотворення даних.

**Перевірка валідації полів**

ОЧІКУВАНА ПОВЕДІНКА:  
Якщо обов’язкові поля не заповнені або містять некоректні дані, система повинна заборонити створення завдання та показати повідомлення про помилку валідації.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 1 секунди.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
100%.

ЗАУВАЖЕННЯ:  
Особливу увагу слід приділити перевірці порожньої назви завдання та некоректної дати.

### **6.3. Тест редагування завдання**

**Коректність зміни даних завдання**

ОЧІКУВАНА ПОВЕДІНКА:  
Після відкриття існуючого завдання, зміни його параметрів і збереження оновлені дані повинні відображатися у списку завдань та у вікні перегляду.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 2 секунд.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
100%.

ЗАУВАЖЕННЯ:  
Необхідно перевірити зміну назви, опису, статусу, пріоритету та дати виконання.

**Збереження цілісності даних після редагування**

ОЧІКУВАНА ПОВЕДІНКА:  
Після редагування повинні змінюватися лише ті поля, які були відкориговані користувачем. Інші дані мають залишатися без змін.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 2 секунд.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
99–100%.

ЗАУВАЖЕННЯ:  
Потрібно перевірити, чи не втрачаються дані внаслідок часткового редагування.

### **6.4. Тест пошуку та фільтрації завдань**

**Коректність пошуку завдань**

ОЧІКУВАНА ПОВЕДІНКА:  
Після введення пошукового запиту система повинна відобразити лише ті завдання, у назві або описі яких присутній відповідний текст.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 1 секунди.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
95–100%.

ЗАУВАЖЕННЯ:  
Слід перевірити пошук за повним словом, частиною слова та відсутнім значенням.

**Коректність фільтрації завдань**

ОЧІКУВАНА ПОВЕДІНКА:  
Після вибору параметрів фільтрації система повинна відобразити лише ті завдання, які відповідають обраному статусу, пріоритету або даті виконання.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 1 секунди.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
95–100%.

ЗАУВАЖЕННЯ:  
Необхідно перевірити як окреме застосування фільтрів, так і їх одночасне комбінування.

### **6.5. Тест видалення завдання**

**Коректність видалення запису**

ОЧІКУВАНА ПОВЕДІНКА:  
Після підтвердження видалення вибране завдання повинно зникнути зі списку, а також бути видаленим із бази даних або позначеним як видалене відповідно до реалізації системи.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 2 секунд.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
100%.

ЗАУВАЖЕННЯ:  
Потрібно перевірити видалення одного завдання та відсутність впливу на інші записи.

**Захист від випадкового видалення**

ОЧІКУВАНА ПОВЕДІНКА:  
Перед видаленням система повинна запросити підтвердження дії. У разі скасування операції завдання має залишитися без змін.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 1 секунди.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
100%.

ЗАУВАЖЕННЯ:  
Перевіряється наявність діалогового вікна або іншого механізму підтвердження.

### **6.6. Тест адаптивності та інтерфейсу**

**Коректність відображення інтерфейсу**

ОЧІКУВАНА ПОВЕДІНКА:  
Усі основні елементи інтерфейсу повинні коректно відображатися на персональному комп’ютері, ноутбуці та мобільному пристрої без накладання, обрізання або втрати функціональності.

ОЧІКУВАНА ШВИДКОДІЯ:  
Сторінка повинна завантажуватися до 3 секунд.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
95–100%.

ЗАУВАЖЕННЯ:  
Перевірка виконується у різних браузерах та на різних роздільних здатностях екрана.

**Зручність взаємодії з інтерфейсом**

ОЧІКУВАНА ПОВЕДІНКА:  
Кнопки, поля вводу, списки та інші елементи керування повинні бути зручними для використання як за допомогою миші та клавіатури, так і через сенсорне керування на мобільних пристроях.

ОЧІКУВАНА ШВИДКОДІЯ:  
Миттєва реакція інтерфейсу або до 1 секунди.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
90–95%.

ЗАУВАЖЕННЯ:  
Оцінка частково має суб’єктивний характер, тому може потребувати додаткового тестування кількома користувачами.

### **6.7. Тест продуктивності та стабільності**

**Швидкість завантаження сторінок**

ОЧІКУВАНА ПОВЕДІНКА:  
Основні сторінки застосунку повинні завантажуватися без помітних затримок, а взаємодія з ними має бути плавною та стабільною.

ОЧІКУВАНА ШВИДКОДІЯ:  
До 2–3 секунд для основних сторінок.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
90–95%.

ЗАУВАЖЕННЯ:  
Перевірка виконується за допомогою Google Lighthouse та засобів браузера.

**Стабільність роботи після внесення змін**

ОЧІКУВАНА ПОВЕДІНКА:  
Після оновлення функціоналу або виправлення помилок система повинна зберігати працездатність раніше реалізованих можливостей без появи нових критичних помилок.

ОЧІКУВАНА ШВИДКОДІЯ:  
У межах базових показників системи.

ОЧІКУВАНА ДОСТОВІРНІСТЬ:  
95–100%.

ЗАУВАЖЕННЯ:  
Для перевірки використовується регресійне тестування із повторним проходженням основних сценаріїв роботи.
